# OpenSensorium 开发清单（Master Checklist）

> 每个需求一个文件，逐条**可验收**。任务按**依赖关系与验证价值**排序（对齐 PRD §11 路线图）。
> 完成一个任务：勾选该任务文件顶部状态 + 更新本表状态 + 按任务 Git 段提交。

**状态图例**：⬜ 未开始 · 🟡 进行中 · ✅ 完成 · ⏸️ 阻塞

**默认环境**：conda `ai_frontiers_env`（Python 3.12，PowerShell）。推理服务跑在 4090 台式机，Core/Endpoint 跑在笔记本，同一局域网。

---

## 开发准则（硬约定，务必遵守）

1. **默认全部未完成**：每个任务、每个验收勾选框初始都是 ⬜。
2. **不许提前标完成**：**只有该需求真正开发并通过验收后**才可把状态改 ✅。未开发时严禁把清单改成已完成。
3. **测试自带**：每个任务由 Agent 自己写**单元测试 / 集成测试 / 回归测试**；能自动判定的先自动判定。
4. **人来把关**：标了 `⚠️ 需人工验证` 的（部署到 4090、真实键鼠/语音/回音、grounding 误差、体感调优等）由**你**验证。每个任务文件都有 `## 分工（Agent 做什么 / 人做什么）`，写明你要做的事与验证方式。
5. **你的反馈优先**：你会逐个审阅任务清单；不符合要求处提出，我据此修改。**最终以你的反馈为准。**
6. **与两款仓库不冲突**：涉及模型接口/流格式处，对齐 MiniCPM-o-Demo 与 JoyAI-VL-Interaction 的真实实现（见 PRD §6.6）。
7. **一般性设计**：面向"未来兼容更多模型"，新增模型只加 Adapter、Core 零改动（FR-2M.5）。
8. **分支 → 你审 → 合并 main**：每个任务在独立分支 `feat/txx-*` 上开发；Agent 完成后推该分支、任务置 🟡。**`main` 是"已认可稳定线"，只有你审阅认可后才把分支合并进 main、任务转 ✅。未过审的代码一律不进 main。** 你在分支/PR 上审阅即为你的把关面。

## 里程碑验收（Demo 级）

- **阶段0 完成**：两款模型服务在 4090 上局域网可达、健康检查通过、Core 能拿到推理结果。
- **M0 完成**：屏幕流稳定进两款模型且不自我回环、UI 可见其所见、一句话切到摄像头、两款视觉路径均跑通并可切换。
- **M1 完成**：像人一样边看边聊、能被打断、会主动开口。
- **M2 完成**：给定具体任务（网页订票/整理文件夹）键鼠闭环完成，危险动作有确认门与急停。
- **M2.5 完成**：把写/改代码结构化外包给文本专家，期间感知不阻塞、结果回流后继续。
- **M2.8 完成**：偏好跨会话生效、定时/条件触发能触发、示范一次后复用一个感知-运动技能。
- **M3 完成**：卡在报错时恰当地主动出手/提示/改，可调不烦人。
- **M4 完成**：人在飞书上让它在电脑上干活并拿到语音/文字回复。
- **M5 完成**：同一套决策既操本地也操云端电脑。
