# OpenSensorium 产品需求文档（PRD v0.2 · Consolidated Draft）

> 状态：草案，用于对齐愿景与范围（已合并多轮讨论收口）
> 日期：2026-07-11
> 一句话定义：**一个以视觉为主、多模态并行的"持续感知—主动行动"通用智能体**。近期以操纵电脑为最低成本验证场，**最终归宿是走进物理世界——操纵汽车、机器人、智能家居等一切可操作之物**。

---

## 0. 名字即纲领

**Sensorium**（感觉中枢）= 一个有机体全部感官的总和及其被整合成统一体验的那个中枢。这是产品的第一性原理：

> 不是"给大模型接几个工具的回合制问答"，而是**给模型一套持续在线的感官与神经系统**——它一直在看、在听，自己决定何时说话、何时动手，并最终**在物理世界中行动**。

所有设计取舍都回到这个隐喻：**传入（afferent，感知）→ 中枢（认知）→ 传出（efferent，行动）**，形成不间断的闭环，而非"问一句答一句"。

## 1. 背景、愿景与问题

### 1.1 现有 Agent 的局限（我们要超越的对象）
以 OpenClaw、Hermes Agent 等为代表的主流范式：
- **文本为主、图像为辅**：世界被压成文字再推理，丢失了信息密度最高的视觉通道。
- **回合制（turn-based）**：用户发问 → 思考 → 回答 → 停止，中间**阻塞**，不主动。
- **单通道串行**：不能"一边看屏幕一边听你说话"。
- **被动**：永远等待触发。

### 1.2 我们的判断：真正像人/动物的智能体
- **实时（real-time）**：感知与响应延迟接近自然交互。
- **非阻塞（non-blocking）**：思考/行动时仍持续感知，可随时被打断（barge-in）。
- **并行（parallel）**：多路感官通道同时在线（视觉+听觉+…）。
- **主动（proactive）**：由内部"注意力/显著性"驱动，自己决定何时开口、何时行动。

视觉是首选主通道（单位时间信息密度最高）；语音次之；触觉、嗅觉等待模型具备后接入。

### 1.3 终极愿景：走进物理世界（务必牢记）
**通过录屏操作手机/电脑，只是第一步，绝非最终产品。** 最终产品是**彻底走进物理世界、与物理世界交互**：
- **操纵汽车**：智能驾驶、智能座舱
- **操作机器人**：工业机器人、家庭机器人
- **操作智能家居**

屏幕操作是通往物理世界的**踏脚石与最低成本验证场**，不是终点。因此本 PRD 的每一个抽象都必须为物理世界执行器保持开放。

### 1.4 底座模型
初期基于具备**实时视觉/语音交互**能力的开源多模态模型：
- [MiniCPM-o](https://github.com/OpenBMB/MiniCPM-o-Demo/)（原生实时视觉+语音双工）
- [JoyAI-VL-Interaction](https://github.com/jd-opensource/JoyAI-VL-Interaction)
- 架构上**不绑定单一模型**：模型是可替换的**推理底座（Model Inference）**，由平台无关的**认知核（Cognition Core，编排逻辑）**经 **Model Adapter** 调用。（"认知核 ≠ 模型"，两者拆分见 §3.6；兼容机制见 FR-2M、§6.6。）

**权重与许可（核对结果，可商用）**：

## 2. 产品定位与差异化

### 2.0 根本分野：抽象符号 vs 直觉形象（第一性差异）
与 OpenClaw / Hermes 等**最本质**的区别，其余差异皆由它派生：

- **它们：符号式、抽象、命题式处理。** 世界先被**串行化成文本/token**，在语言这个**强制中介瓶颈**上推理，再吐文本动作。有损压缩、偏慢、偏审慎（近似 System 2）。
- **OpenSensorium：感知式、具身、直觉形象处理。** 世界经感官（视/听/嗅/触）**被直接感知**，以**直觉的、形象的（analog / imagistic）**方式处理并直接转化为行动。**语言不是必经中介**——感知是第一性的、直达的。近似 System 1 / 感觉-运动认知，是人和动物真正在世界里行动的方式。

> 一句话：**它们"把世界读成一段文字再思考"；我们"直接看着世界、凭直觉行动"。**
> （注：这不是排斥语言推理，而是拒绝让语言成为感知与行动的**强制中介**。当确需精确符号推理时，走"认知委派"外包给文本专家，见 §3.5。）

### 2.1 差异化对照

| 维度 | 传统文本 Agent | **OpenSensorium** |
|---|---|---|
| **信息处理方式** | 抽象符号 / 命题式（先译成文字） | **直觉形象 / 具身感知（直接感知与行动）** |
| 主感知通道 | 文本 | **视觉**（+ 语音，+ 未来触觉/嗅觉） |
| 交互模式 | 回合制、阻塞 | **持续、非阻塞、可打断** |
| 通道并行 | 串行 | **多路感官并行** |
| 主动性 | 被动应答 | **显著性驱动的主动交互** |
| **行动/操作** | 软件工具调用（弱执行） | **一等公民**：键鼠 → 云端电脑 → 手机/汽车/机器人/家居（**可扩展执行器抽象**） |
| 输入 | 文本/图片 | 视频流、音频流、文本、图片、视频 |
| 输出 | 文本 | 文本 + 语音（+ 物理动作） |

**产品愿景（长期）**：像生物一样持续感知物理与数字世界、并主动在其中行动的通用智能体。
**产品定位（近期，本 PRD 聚焦）**：一个**持续观察电脑屏幕（及麦克风）、以键鼠操作电脑、可通过语音/飞书与人交互的实时多模态 Agent**，作为通往物理世界的第一步。

## 3. 设计哲学（贯穿全文的核心决策）

### 3.1 感知面 vs 呈现面 解耦（解决"视觉回音"）
把屏幕流传入模型、同时又把 OpenSensorium 界面显示在屏幕上，会产生**视觉版声学回声**：模型看到自己的输出 → 正反馈发散。

**决策：显式区分两个表面**
- **Observation Surface（被观察面）**：模型实际采集的内容（目标应用/显示器/摄像头）。
- **Presentation Surface（呈现面）**：OpenSensorium 自己的 UI、self-view、结论浮层。

**回音判定的一般准则**：**回音风险存在 ⟺ 呈现面可能出现在被采集的观察面之内。** 这与"本地/云端"无关，只取决于**内容渲染在哪块显示屏、是否与 OpenSensorium 的 UI 叠在同一屏被采集**。
- **需要处理**：观察面与呈现面在**同一块被采集的显示屏/窗口**上可能重叠时（如本地屏自采集；或云端电脑经 RDP/VNC **渲染到本地屏**、本地又叠了浮层——见 §9.3）。
- **无需处理**：观察面与呈现面**物理不同屏**、不可能重叠时（如摄像头源；或云端电脑**不在本地屏渲染**、云端侧直接抓帧的无头形态）。

借鉴视频会议"既录屏又显示"的三招，定分级方案：
1. **源隔离**：采集"某窗口/某显示器"而非整块合成屏；UI 放到另一显示器/虚拟桌面。
2. **采集级排除（Windows 首选，最优雅）**：`SetWindowDisplayAffinity(hwnd, WDA_EXCLUDEFROMCAPTURE)` 让浮层**对人眼可见、对采集隐形**；macOS 用 ScreenCaptureKit `contentFilter` 排除自身窗口。
3. **虚拟显示器**：给 Agent 独立虚拟屏操作，人在主屏观看，物理上不可能回环。

> **近期落地**：Windows"窗口级采集 + `WDA_EXCLUDEFROMCAPTURE`"即可干净解决；强隔离再上虚拟显示器。仅"屏幕自采集"这一路需处理，摄像头等其它源无回环。
> **跨平台**：各系统的"采集级排除"能力差异很大（Windows/macOS 原生支持按窗口排除，Linux/Android/iOS 不支持），需按平台退化到源隔离或虚拟显示器方案，详见 §6.4 平台兼容矩阵。

### 3.2 传入—中枢—传出 三通路（行动与感知并列）
- **Afferent（传入/感知层）**：各路流规整成带时间戳的"感知事件"。
- **Cognition Core（中枢/认知层）**：持续消费感知事件，维护工作记忆与注意力，产出"意图"。
- **Efferent（传出/行动层）**：把意图翻译成执行器动作。

> **行动是与感知并列的第一性能力，绝不次要。** 感知的终点是**改变世界**；只会看、会说却不会动手的智能体是残缺的。

### 3.3 执行器抽象与"预留开放"
所有行动走统一的 **Effector 接口**，经**能力协商（capability negotiation）**声明能做什么。今天注册键鼠/语音执行器，未来直接注册云端电脑/手机/汽车/机器人/家居执行器，认知核无需改动。（详见 §9）

### 3.4 安全优先（因为它真的会动手）
它操作真实电脑、未来操作物理世界，**误动作代价高（物理世界尤甚）**。从第一天起就要有：动作确认门、危险动作白/黑名单、全局急停（kill switch）、可回放审计、作用域边界。（详见 §8）

### 3.5 异构认知分工：感知核编排文本专家（认知委派）
**问题**：视觉优先的感知核在**文本空间的精确操作**（写正确代码、长程符号推理）上偏弱——这是"直觉形象处理"（§2.0）的固有代价。

**决策**：不要求感知核什么都强，让它**编排专家**——从框架层第一天起内建**认知委派（Cognitive Delegate）**，把文本/符号任务委派给专门文本智能体（**Codex、Claude Code** 等）。
- 类比：感知核 ≈ 感觉-运动皮层；编码智能体 ≈ 可调用的语言/符号推理模块（异构认知分工）。
- **"用外部编码工具"有两条一等路径，都保留**（这是两种机制，不是二选一淘汰）：
  1. **结构化委派（Delegate，"用别人的脑子"）**：把任务交给另一个文本智能体（sub-agent / API / CLI / MCP），结构化传入上下文、结构化拿回结果（patch/日志/成败）。
  2. **屏幕操作该工具（Effector，"用自己的手脚"）**：OpenSensorium 用键鼠 + 感知回路，像人一样打开并操作 **ClaudeCode / Codex / Cursor** 等工具（点快捷键→输入→看输出）。这在概念上属**原生 GUI 操作**（§3.2/§9），术语上**不叫"委派"**，但它是达成同一目的的合法路径。
- **两条路径怎么选**：
  - 结构化委派胜在**异步、非阻塞、可传大上下文**——派出任务后感知回路继续跑，契合"非阻塞、并行"（§1.2）。
  - 屏幕操作用于：**工具无可用 API**；或**该工具本质是 GUI（如 Cursor 这类 IDE），像人一样用它更自然**；代价是会占用主回路（盯屏）。
- **术语守恒**：无论走哪条路径，"委派(Delegate)"一词仍**只指结构化那条**；屏幕操作始终归 Effector。详见 §9.6。

### 3.6 三件套拆分：Endpoint / Core / Inference（跨平台兼容的架构基石）
要同时兼容 **Windows / macOS / Linux / Android / iOS**，把系统拆成三个可独立部署的部分：

- **Endpoint（端，每平台一个薄适配器）**：负责该系统上的**采集（屏幕/摄像头/麦克风）+ 执行（键鼠/触屏/…）+ 权限**，把感知流上行、把动作指令下行到本机。端是"感官+手脚"，尽量薄。
- **Cognition Core（核，平台无关，硬件无关）**：认知核编排逻辑 + 感知总线 + 工作记忆 + 意图/委派编排。**不必坐在 GPU 上**——它只需能**访问一个模型推理 API**。一份代码不随平台变，**任何硬件/平台都能跑**（含 CPU-only 的小机器，甚至理论上与某个 Endpoint 同机）。
- **Model Inference（模型推理服务，吃 GPU）**：真正做前向计算的 VLM/多模态模型，以 **API 形式**对外提供。宿主可为：**本机 GPU / 局域网 GPU 机 / 远程或云 GPU API**——**任意可达形态皆可**。"本地自托管"指模型服务归用户掌控，不等于必须跑在本地 GPU 硬件上。

**关键推论**：
- Core 与 GPU 解耦 → **算力可弹性**：低配设备跑 Core、把推理指向远程 GPU；高配设备本地一体化。
- 手机（尤其 iOS）跑不动本地推理 → 手机永远是 **Endpoint**，Core + 推理在别处，经实时流连接。这也统一了"操作手机"场景（§9.4.1）。
- 每平台的采集/防回音/输入注入差异全部收敛在 Endpoint，Core 与 Inference 无感。详见 §6.4。

### 3.7 主动性的三个触发源（定时任务是其中之一）
"主动交互"（§1.2）不是只靠感知。它由三类触发源共同驱动，缺一不可：
1. **感知触发（Salience）**：看到/听到显著变化 → 主动出声或行动（已在 §3.2/FR-2.3）。
2. **时间触发（Schedule）**：**定时任务 / cron / 一次性提醒** → 到点主动执行（如"每天9点汇报""10分钟后提醒我"）。
3. **事件触发（Event）**：channel 来消息、Delegate 结果回流、外部 webhook 等。

> 这正是把 OpenClaw 式的**定时任务机制**接入本产品的位置——它是主动性的"时间"那一维。三类触发统一进入认知核，由显著性/优先级仲裁"此刻做什么"。

### 3.8 记忆机制：持续感知必须有长期记忆
一个**常开、一直在看**的智能体会产生海量经验；只有滚动的工作记忆（FR-2.2）等于"金鱼脑"，一滚动就忘。因此需要分层记忆 + 巩固（consolidation）：**工作记忆 → 情景记忆 → 语义/长期记忆 → 程序性记忆（技能）**，可检索回流到认知核（详见 §6.5、FR-7）。
> 记忆同时带来**隐私责任**：常开感知会记录大量内容，记忆需有**保留期、脱敏、用户可查可删**策略（并入 Safety，§8）。

## 4. 用户与核心场景

### 4.1 目标用户（近期）
- 开发者/极客：想要"看着我屏幕、主动帮我干活/提醒/动手"的实时助手。
- 需要 GUI 自动化但讨厌脆弱脚本的人：用视觉理解代替写死的坐标脚本。

### 4.2 核心场景（v1 聚焦电脑操纵，四类都在路线图里）
> **定位强调：操作/行动是与感知并列的第一性能力，绝不次要。** 四类场景里"能动手操作"始终是核心衡量标准。

1. **看屏 + 键鼠完成单个任务**（GUI Agent 闭环）
   例："帮我在这个网页上订张票""把这个文件夹按类型整理"。模型看屏 → 决策 → 输出鼠标坐标+点击/键盘 → 改变环境 → 再看屏。**这是产品的看家能力。**
2. **主动陪伴/观察式助手（观察 + 随时上手操作）**
   持续看屏，需要时主动出声/提示（体现"非阻塞、主动"）；**一旦你同意就立刻上手操作**。例：看到你卡在报错上，主动说"这个报错我见过，我直接帮你改？"——得许可即动手。观察与操作连续，不是"只观察不操作"。
3. **飞书为入口的交互 Agent**
   通过飞书收发消息/语音/图片，把**"屏幕操作"这一核心能力**作为后端被远程调用——重点是远程让它"干活/动手"，而非只远程问答。
4. **实时视觉+语音双模对话（边看边聊，且能随时转为动手）**
   "像人一样边看边聊"；对话中随时可转入操作。**操作能力同样是一等公民**，语音对话是交互皮肤，动手改变环境才是价值落点。

### 4.3 关键体验要求
- **可随时打断（barge-in）**：它在说话/操作时，你一开口它就停下听你。
- **主动但不烦**：显著性阈值可调，避免话痨。
- **可见其所见**：呈现面让人看到"它现在在看什么、准备做什么"，建立信任。

## 5. 范围（Scope）

### 5.1 v1 In-Scope（基准假设：自托管推理[GPU 可本地/远程] / Core 硬件无关 / 持续常开感知 / 视觉+语音并列 / **单路视频流 + 可切源**）
- **单路视频流**实时采集：某一路——屏幕 **或** 摄像头（前置/后置）——一次只接一路；v1 以屏幕为主。
- **视频源运行时热切换**：一句话在 屏幕 / 前置 / 后置摄像头 之间切换当前这一路（见 FR-1.6）。
- 屏幕这一路：感知面/呈现面解耦（防回音）；摄像头源无回环，不需此处理。
- 麦克风音频流实时采集 + VAD + 语音输入。
- **模型完全兼容 MiniCPM-o 与 JoyAI-VL 两者**（经 Model Adapter + 能力协商，非二选一；未来更多模型只加 Adapter，见 FR-2M）。
- 认知核持续感知回路（常开），显著性驱动的主动发声/行动。
- 键鼠执行器（屏幕坐标 + 点击/拖拽/键入）——**操作能力为核心**。
- 用外部编码工具（Codex / ClaudeCode / Cursor）**两条路径都保留**：① 结构化委派（sub-agent/API/CLI/MCP，异步/非阻塞）② 屏幕操作该工具（Effector，像人一样开窗操作）；按场景选，见 §3.5。
- 语音输出（TTS）+ 文本输出；可打断（barge-in）。
- 飞书 channel（收发文本/语音/图片）。
- **通用 Agent 能力**：分层记忆（工作/情景/语义/程序）+ 巩固检索、定时任务/cron/条件触发调度、长时任务断点续跑、技能注册表（符号工具 + 感知-运动技能，含示范学习雏形）。
- 本地叠加式 UI（呈现面）：显示"它在看什么、结论、动作日志"。
- 安全：动作确认门 + 全局急停 + 操作审计回放 + 作用域边界 + 记忆隐私。

### 5.2 Out-of-Scope（v1 明确不做，但架构预留）
- **多路视频流**（如屏幕+摄像头同时）。⚠️ **受底座模型能力限制**——当前模型尚不支持多路视频流同时接入，须**待模型具备后**才做；架构上感知事件总线以 source id + 时间戳为多路预留。
- 云端电脑、手机/汽车/机器人/家居等**本地电脑之外的执行器**（Effector 接口预留，见 §9.3 演进路径）。
- 触觉/嗅觉输入（感知层预留通道类型）。
- 微信及其他 channel（Channel 接口预留）。
- 多 Agent 协同、云端多租户。

### 5.3 未来（Vision，非承诺）
多路视频流融合（待模型具备）、云端电脑及**物理世界执行器（汽车/机器人/家居）**、更多感官通道、更多 channel。

## 6. 系统架构

> 三件套：**Endpoint（每平台薄端）↔ Core（硬件无关编排）↔ Inference（吃 GPU，API）**。感知→认知→行动闭环 + Delegate 支路。

```
┌── ENDPOINT (每平台薄端: Win/mac/Linux/Android/iOS) ─────────────────────────┐
│  采集: 屏幕 / 摄像头 (单路+可切源) / 麦克风 / 文本·图片·视频               │
│  呈现: 本地UI · self-view · 结论浮层   ── 感知面/呈现面解耦(防回音)         │
│  执行: 键鼠 / 触屏 / (未)手机·车·机器人·家居                                │
└──────▲────────────────────────────────────────────────────┬───────────────┘
  感知流 上行│  JPEG ~1fps(可10fps) · 16kHz mono PCM16 ~1s · 带时间戳         │动作指令 下行
  (WebRTC/WS)│  ▲切源等"控制指令"(独立控制面)                                 │
┌──────┴──── CORE (硬件无关, 只需访问推理API) ─────────────────────────────────▼──┐
│  感知事件总线(统一时间戳, 跨模态对齐)                                            │
│  认知核: 工作记忆 · 注意力/显著性 · 可打断(barge-in) → 意图{说/做/委派/静默}     │
│  通用能力层: 记忆(工作·情景·语义·程序) · 调度(cron/条件) · 任务(断点续跑) · 技能  │
│  主动性三触发源: 感知(salience) + 时间(定时任务) + 事件(channel/委派回流)         │
│  Safety: 确认门·急停·审计·作用域·记忆隐私 (一切动作/委派经此)                    │
└──┬────────────────┬─────────────────────┬──────────────────────┬──────────────┘
   │意图→动作         │委派(仅结构化)          │访问推理 API            │Channel
   ▼                 ▼                     ▼                      ▼
[Efferent 行动层]  [Delegate 认知委派]   ┌── INFERENCE (吃GPU: 本地/局域网/云) ──┐  [飞书 /(未)微信]
 键鼠·触屏·语音·文本  Codex/ClaudeCode     │ Model Adapter (能力协商)              │   ↕ 远程用户
 (未)物理世界执行器   sub-agent/API/MCP    │  ├ MiniCPM-o : 原生语音(音频进/出,双工)│
   │  "用自己的手脚"  异步·"用别人的脑子"   │  └ JoyAI-VL  : VL + 外挂ASR/TTS/bg-agent│
   │                                     └──────────────────────────────────────┘
   ▼ 动作作用于环境
[ 本地电脑 → 云端电脑 → 手机 → 汽车/机器人/智能家居 (物理世界·终极目标) ]
   └──────────── 环境变化被 Endpoint 再次采集 → 回到感知总线 (闭环) ────────────┘
```

### 6.1 分层职责
| 层 | 归属 | 职责 | v1 组件 | 预留扩展 |
|---|---|---|---|---|
| **Afferent 感知层** | Endpoint | 各路流采集、采样、规整成带时间戳感知事件；音频 VAD；**单路+运行时切源** | 屏幕/摄像头/麦克风采集器、文件导入 | 多路并行（待模型）、触觉、嗅觉 |
| **Surface 管理** | Endpoint | 感知面/呈现面解耦，防回音 | 窗口级采集 + `WDA_EXCLUDEFROMCAPTURE` | 虚拟显示器、多屏 |
| **Efferent 行动层** | Endpoint | 意图→执行器动作，能力协商 | 键鼠、触屏、语音(TTS)、文本 | 云端电脑、手机、汽车、机器人、家居 |
| **Cognition Core 认知层** | Core | 持续感知、工作记忆、注意力/显著性、产出意图；可打断；**硬件无关** | 感知总线 + 编排逻辑 | 可换/多模型路由 |
| **Model Inference/Adapter** | Inference | 模型前向计算(吃GPU)；Adapter 封装+能力协商 | MiniCPM-o、JoyAI-VL 两个 Adapter | 更多实时多模态模型 |
| **Delegate 认知委派** | Core | 结构化外包文本/符号子任务，异步回收 | Codex、Claude Code | 更多文本智能体 |
| **Memory 记忆层** | Core | 分层记忆 + 巩固 + 检索回流 | 工作/情景/语义/程序记忆、向量+结构化存储 | 跨设备同步、知识库 |
| **Scheduler 调度层** | Core | 定时任务/cron/一次性提醒/条件触发 | 时间触发器、条件触发器 | 分布式调度 |
| **Task/Goal 任务层** | Core | 长时目标与待办的跟踪、恢复、优先级 | 任务状态机、断点续跑 | 多目标编排 |
| **Skills 技能层** | Core | 可注册的工具/技能（含内置能力） | 技能注册表、能力协商 | 技能市场/自学技能 |
| **Channel 层** | Core | 与远程用户收发通道 | 飞书、本地 UI | 微信、其他 IM |
| **Safety 层** | Core | 确认门、急停、审计、作用域、**记忆隐私（保留期/脱敏/可删）** | 全部动作经此层 | 物理动作分级授权 |

### 6.2 实时数据面
- **感知事件总线**：所有感知层输出统一时间戳事件（视频帧、音频片段、文本、Delegate 结果回流…），认知核订阅消费。
- **控制面 vs 行动面分离**：用户"切源/停止"等**控制指令**走独立控制面，与"行动意图"分开。
- **传输**：本地进程内优先低延迟通道；跨端（飞书/远程观看/Endpoint↔Core）用 WebRTC 类实时传输承载音视频。
- **背压与采样**：视频流按显著性动态调采样率（静止降帧、变化升帧），控制负载与成本。

### 6.3 Endpoint / Core / Inference 部署形态
- **Model Inference（推理服务）**：吃 GPU，以 API 暴露。宿主任意可达形态——本机 GPU / 局域网 GPU 机 / 远程或云 GPU API。**经 Model Adapter 兼容多款模型**：v1 完全兼容 **MiniCPM-o 与 JoyAI-VL 两者**，每款一个 Adapter 声明其实时能力剖面，Core 按能力协商适配；未来更多实时多模态模型只加 Adapter（见 FR-2M）。
- **Core（认知核）**：平台无关、**硬件无关**，只需能访问推理 API。可跑在任意设备（含 CPU-only）。
- **Endpoint（每平台薄端）**：桌面端（Win/mac/Linux）可与 Core **同机**（本地进程内，最低延迟）；移动端（Android/iOS）**必为远程端**，经实时流连 Core。
- **三者可自由组合**：一体机（Endpoint+Core+本地GPU推理同机）｜瘦端+云脑（Endpoint 在设备、Core+推理在云）｜Core 在本地/推理在远程 GPU（低配设备也能用）等。
- **端能力协商**：每个 Endpoint 声明本平台可用的采集源、防回音手段、可注入动作原语、权限状态；Core 据此裁剪可行动作（如 iOS 端注入受限时自动降级或提示）。
- **开发语言：Python 主力**。Core 与桌面 Endpoint（Win/mac/Linux）以 Python 为主（契合 MiniCPM-o/JoyAI 的 Python+vLLM/PyTorch 栈）；移动端等 Python 覆盖不到的采集/注入再用平台原生语言写薄壳，经统一实时流接回 Python Core（见 §12.10）。

### 6.4 平台兼容矩阵（Windows / macOS / Linux / Android / iOS）
> 兼容目标：Core 一份代码（硬件无关）；差异全部收敛在 Endpoint。以下为各平台采集 / 防回音排除 / 输入注入 / 本地推理能力的选型与约束。

| 平台 | 屏幕采集 | 摄像头/麦克风 | 防回音（采集排除） | 输入注入（执行器） | 本地跑 GPU 推理？ | 权限要点 |
|---|---|---|---|---|---|---|
| **Windows** | Windows.Graphics.Capture / DXGI Desktop Duplication | Media Foundation | ✅ **原生按窗口**：`WDA_EXCLUDEFROMCAPTURE` | `SendInput` / UI Automation | ✅（有 GPU 时）；否则指向远程推理 API | 无特殊 |
| **macOS** | ScreenCaptureKit（12.3+） | AVFoundation | ✅ **原生按窗口**：`SCContentFilter` 排除自身 | `CGEventPost`（需**辅助功能**授权） | ✅（Apple Silicon）；否则远程 | 屏幕录制 + 辅助功能授权 |
| **Linux** | PipeWire + xdg-portal（Wayland）/ X11(XShm) | V4L2 / PipeWire | ⚠️ 无按窗口排除 → **源隔离 / 虚拟显示器**（Xvfb/虚拟输出） | Wayland：libei/RemoteDesktop portal；X11：XTEST；底层 uinput | ✅（有 GPU 时）；否则远程 | portal 授权（Wayland 较严） |
| **Android** | MediaProjection API | Camera2 / AudioRecord | ⚠️ 无按窗口排除 → **单应用采集（A14+）/ 独立虚拟 Display / 分屏隔离** | **AccessibilityService**（dispatchGesture：tap/swipe）或 ADB `input` | ❌ → 推理走远程 API（Core 仍可在端上） | 录屏授权 + 无障碍服务授权 |
| **iOS** | ReplayKit（应用内/广播扩展） | AVFoundation | ⚠️ 无排除、沙盒最严 → **源隔离 / 跨设备或云设备** | **无公开跨应用注入**：XCUITest/WebDriverAgent（开发态）、辅助触控/切换控制、或云真机 | ❌ → 推理走远程 API | 沙盒限制强，跨应用自动化受限 |

**由矩阵得出的产品决策**：
- **v1 首平台 = Windows**（防回音与输入注入最顺，且是你当前环境）。桌面三平台（Win/mac/Linux）为 v1~v1.x 目标；移动端（Android/iOS）随"操作手机"里程碑（§9.4.1、M6+）以 Endpoint 形态接入。
- **防回音分级**：能原生排除的（Win/mac）用排除；不能的（Linux/Android/iOS）退化到**源隔离或虚拟显示器**——这正是 §3.1 三招分级的用途。
- **iOS 是最受限平台**：跨应用输入注入无公开通路，"操作 iOS"近期只能靠开发态自动化（WebDriverAgent）或**云真机**；感知（录屏/摄像头）相对可行。此约束需在 §12 单列。
- **推理与硬件解耦**：任何平台**本地无 GPU 都不阻塞使用**——把推理指向远程/云 GPU API 即可；Core 硬件无关（§3.6）。本地有 GPU 则可一体化低延迟。

### 6.5 通用 Agent 能力层（记忆 / 调度 / 任务 / 技能）
除感知—认知—行动主回路外，还需一套与 OpenClaw 等通用 Agent 对齐的**认知基础设施**，挂在认知核旁：

- **Memory 记忆**（见 §3.8）：
  - **工作记忆**：滚动短期上下文（当前屏幕、对话、任务、视觉源）。
  - **情景记忆（episodic）**：发生过什么——会话/事件流，可回放（与审计共用底座）。
  - **语义/长期记忆（semantic）**：事实、用户偏好、习得知识；向量检索 + 结构化存储。
  - **程序性记忆（procedural）**：习得的"怎么做某任务"，喂给 Skills 层复用。
  - **巩固（consolidation）**：把工作记忆里值得留存的经验后台提炼进情景/语义记忆。
- **Scheduler 调度**（见 §3.7）：定时任务、cron、一次性提醒、**条件触发**（"当你看到构建失败，就通知我"）。到点/触发产生一个"内部事件"进入认知核。
- **Task/Goal 任务目标**：管理长时目标与待办；支持**断点续跑**（常开 Agent 可能被打断/切源/重启，任务状态要能恢复）；优先级仲裁并发目标。
- **Skills/Tools 技能**：**技能形态在感知式 Agent 里发生了根本变化，分裂为两类**：
  - **符号工具技能（symbolic tool）**：传统形态——有文本 schema（名字/参数/描述）的函数或 API 调用，由开发者编写、显式调用。复杂的符号任务往往转交 Delegate（§9.6）。
  - **感知-运动技能（perceptual-motor skill）**：**新形态、且是主体**——不是 JSON 函数，而是一段"感知→行动"的过程性策略（看→动→验证），如"登录这个 App""关掉这个弹窗"。它更像人的肌肉记忆/习惯，落在**程序性记忆**里，**靠示范与经验习得**而非写死 schema；键控的是"东西长什么样"而非 API 契约。
  - 详见下文"技能形态为何巨变"讨论（回应设计问题）。

> 与主线的关系：这些是"神经系统旁的海马体/生物钟/习惯"。**记忆让它有连续的自我，调度让它有时间维度的主动，任务让它能坚持长目标，技能让它可积累能力。**

#### 6.5.1 技能形态为何巨变（感知式 vs 文本式）
| | 文本 Agent 的技能 | **OpenSensorium 的技能** |
|---|---|---|
| 本质 | 文本 schema 的函数调用（符号） | **感知-运动过程策略**为主 + 符号工具为辅 |
| 定义方式 | 开发者写 schema | **示范/经验习得**为主（程序性记忆巩固），也可开发者内置 |
| 调用方式 | 显式发 `tool_call(...)` | 多数**融入连续行动流**直接"做出来"，技能边界更模糊 |
| 依赖 | API 契约 | **界面长什么样**（视觉锚点），跨 App 泛化逻辑不同 |
| 鲁棒性 | API 变即失效 | 视觉理解对 UI 微调更适应，但对视觉大改更敏感 |
| 与记忆关系 | 无（外部工具） | **技能 = 程序性记忆的产物**，会"越用越会" |

> 结论：**是的，技能会大变**。产品要把"技能"设计成一个**从符号工具到感知-运动过程的谱系**，且核心可通过**示范学习 + 程序性记忆巩固**来积累，而不是只提供"写 schema 注册工具"这一条路。

### 6.6 感知流格式基线与两款模型对齐（据仓库文档核对）
> 目的：确保 Afferent/Efferent 的流格式与 **MiniCPM-o**、**JoyAI-VL** 均兼容。以下为核对其仓库/文档后的事实与由此定的基线。

**两款模型的实时接口事实**：

| 维度 | MiniCPM-o（omni 端到端） | JoyAI-VL（VL + 可插拔服务，vLLM） |
|---|---|---|
| 音频输入 | **原生**：16kHz 单声道 PCM、1s/块（16000 采样，`MIN_AUDIO_SAMPLES=16000`），waveform | **需外部 ASR**：PCM16 → Qwen3-ASR → 文本，再入 VL |
| 语音输出 | **原生 TTS**（模型出音频 token；音色 ref_audio 16kHz mono） | **需外部 TTS**：文本 → Qwen3-TTS(vLLM-Omni) → PCM16 |
| 视频输入 | PIL 图，≤~180万像素，≤10fps，`stack_frames`(主帧+子帧) | **JPEG 帧 ~1fps**，WebRTC→webinfer HTTP:8070 |
| 会话/推理 | `streaming_prefill` + `streaming_generate`（全双工 TDM 时间片交织） | **OpenAI 兼容 HTTP API** + vLLM |
| 主动/双工 | 模型内全双工，**~1Hz 自主决定是否说话** | 每帧输出信号：`</silence>` / `</response> 文本` / `</delegate> 任务` |
| 传输 | WebSocket（gateway→worker） | WebRTC(视频) + WebSocket(ASR/TTS) + HTTP(推理) |

**关键分歧 → Adapter 必须抽象掉的两点**（落到 FR-2M.3 能力协商）：
1. **语音架构**：`native-speech`（MiniCPM-o：原生音频进/出）vs `external-asr-tts`（JoyAI：外挂 ASR/TTS 管线）。→ OpenSensorium 音频输入(FR-1.3)与语音输出(FR-3.2)对上层统一，具体由 Adapter 决定"直接喂音频/出音频" 还是"先 ASR→文本、文本→TTS"。JoyAI 的 Adapter 需一并封装 ASR/TTS/background-agent 服务。
2. **主动/委派信号**：JoyAI 的 `</silence>/</response>/</delegate>` **恰好映射**到本产品的 显著性静默 / 主动发声（§3.7、FR-2.3）/ 认知委派（§9.6）；MiniCPM-o 则以 1Hz 自主发声体现。→ Adapter 把各自的"是否说话/是否委派"信号归一成 Core 的统一意图（FR-2.5）。

**兼容两款的感知流格式基线（取公约数，Afferent 产出即满足两者）**：
- **音频**：**16kHz 单声道 PCM16，~1s 分块**（附时间戳）。
- **视频**：**JPEG 帧，默认 ~1fps（可上探至 10fps），分辨率下采样至 ≤~180万像素**，每帧为独立静止图。
- **时间对齐**：音视频块带统一时间戳（FR-1.5），契合 MiniCPM-o 的 TDM 交织。
- **传输**：Endpoint↔Core 用 WebRTC 类实时流承载上述音视频（与两款 demo 传输一致）。

> 结论：**当前 PRD 的流格式与两款模型兼容**，代价是 Adapter 层需吸收"原生语音 vs 外挂 ASR/TTS"的分歧。此分歧已在 §12 单列为需拍板项（JoyAI 需额外部署 ASR/TTS 服务）。

## 7. 关键功能需求（FR）

### FR-1 感知层
- FR-1.1 屏幕采集：全屏/指定显示器/指定窗口三种源；可配采样率与分辨率下采样。**帧格式基线（兼容两款模型，见 §6.6）**：JPEG 静止帧、默认 ~1fps（可上探 10fps）、分辨率下采样至 ≤~180万像素。
- FR-1.2 防回音：呈现面窗口默认对采集隐形（`WDA_EXCLUDEFROMCAPTURE`）。
- FR-1.3 音频采集：麦克风流 + VAD，输出语音片段事件。**格式基线**：16kHz 单声道 PCM16、~1s 分块、带时间戳（兼容 MiniCPM-o 原生音频与 JoyAI 的 ASR，见 §6.6）。
- FR-1.4 其它输入：文本、图片、视频文件作为一次性感知事件注入。
- FR-1.5 每个感知事件带**统一时间戳**，供跨模态对齐。
- FR-1.6 **视频源运行时热切换（Source Switching）**：单路视频流绑定的源可运行时切换——屏幕 ↔ 前置 ↔ 后置摄像头。
  - 由用户**控制指令**触发（语音/文本/飞书，如"切到前置摄像头"），走独立控制面。
  - 切源须无缝：旧源停采、新源起采、工作记忆标注"视觉源已从X切到Y"，避免混为同一场景。
  - 语义是**单路 + 可切源**，非多路同时；架构为未来多路预留。

### FR-2 认知核
- FR-2.1 **持续感知回路**（常开）：不依赖用户触发。
- FR-2.2 **工作记忆**：滚动上下文（屏幕状态、对话历史、当前任务、当前视觉源）。
- FR-2.3 **注意力/显著性**：对"何时值得行动/开口"打分，超阈值才主动；阈值可配（防话痨）。
- FR-2.4 **可打断（barge-in）**：用户开口/发消息时立即中止当前生成或动作，转入倾听。
- FR-2.5 **意图输出**：结构化意图（说什么 / 做什么动作 / 委派什么 / 什么都不做）。

### FR-2M 模型兼容（Model Adapter）
- FR-2M.1 **v1 完全兼容两款模型**：**MiniCPM-o 与 JoyAI-VL 均为一等兼容目标**（不是二选一），两者的模型服务都要能作为认知核的底座跑通全部核心回路。
- FR-2M.2 **模型适配器抽象**：每款模型一个 Adapter，把该模型的接口/流式协议/token 化封装到 Core 的统一接口后面；Core 逻辑不随模型变。
- FR-2M.3 **能力协商**：Adapter 声明该模型支持的实时能力（视觉流、语音输入、语音双工/TTS、可打断、上下文长度、单/多路视频…）；Core 据此启用/降级对应功能（与 Effector/Endpoint 能力协商同构）。**其中关键的一维是语音架构：`native-speech`（如 MiniCPM-o，原生音频进/出）vs `external-asr-tts`（如 JoyAI-VL，需外挂 Qwen3-ASR/Qwen3-TTS 与 background-agent）**——JoyAI 的 Adapter 需一并封装这些服务，见 §6.6。
- FR-2M.4 **运行时可切换**：可在 MiniCPM-o 与 JoyAI-VL 间切换，甚至按任务路由到不同模型。
- FR-2M.5 **面向未来开放**：新增实时多模态交互模型只需实现新 Adapter，Core 与上层零改动。

### FR-3 行动层（Efferent）
> 动作原语词汇现在就完整定义（键鼠 + 触屏）；具体执行器按里程碑交付——键鼠随 v1，触屏随"操作手机"里程碑（§9.4.1、M6+）。
- FR-3.1 **键鼠执行器**（本地电脑 v1）：绝对/相对坐标移动、左右键点击、双击、拖拽、滚轮、键盘输入、组合键。
- FR-3.1T **触屏执行器**（手机等触屏设备，随手机里程碑）：`touch.tap`（点击）、`touch.long_press`（长按）、`touch.long_press_drag`（长按拖拽）、`touch.swipe`（左滑/右滑/上滑/下滑，含起止坐标与速度）、`touch.drag`、`touch.pinch`（缩放）、`key.back/home`（系统键）、文本输入。（与 §9.4.1 PhoneEffector 一致）
- FR-3.2 语音输出（TTS）：可打断的流式语音。**由 Adapter 决定实现**：`native-speech` 模型（MiniCPM-o）直接出音频；`external-asr-tts` 模型（JoyAI-VL）经外部 TTS（Qwen3-TTS）合成（见 §6.6、FR-2M.3）。
- FR-3.3 文本输出：面向本地 UI / 飞书。
- FR-3.4 **Effector 能力协商**：每个执行器声明动作原语；认知核只发"意图"，行动层匹配可用执行器。
- FR-3.5 所有动作先过 Safety 层（见 FR-6）。
- FR-3.6 **动作接地与校验**（GUI 闭环的核心一环）：(a) **坐标接地**——模型输出的点击坐标要对齐到目标元素，必要时叠加元素检测/OCR 等辅助定位；(b) **执行后校验**——由下一帧感知确认动作是否生效，未生效则重试/换策略（这是"看家能力"可靠性的关键，参见 §12.14）。

### FR-3D 认知委派（Cognitive Delegate，弥补文本空间短板）
- FR-3D.1 **委派编码/符号任务**：`code.write / code.fix / repo.refactor / longform.reason` 等委派给文本智能体（v1：Codex、Claude Code）。
- FR-3D.2 **委派(Delegate)术语只指结构化一种**：经 sub-agent / API / CLI / MCP 传入上下文、拿回结构化结果（patch/日志/成败）。**但"用外部编码工具"保留两条一等路径**：本结构化委派 + 屏幕操作该工具（Effector，FR-3.1，如开窗操作 ClaudeCode/Codex/Cursor）；两条按场景选，见 §3.5/§9.6。
- FR-3D.3 **异步非阻塞**：委派为异步派发，感知回路不阻塞；结果到达作为感知事件回流。（这正是相对"自己在屏幕上操作工具"的价值所在。）
- FR-3D.4 **可插拔**：新增/更换文本智能体只需注册新 Delegate，认知核零改动。
- FR-3D.5 Delegate 产生的实际动作（改文件/跑命令）同样过 Safety 层。

### FR-4 Channel（飞书优先）
- FR-4.1 飞书：接收文本/语音/图片，发送文本/语音回复。
- FR-4.2 会话与身份：区分用户/群；权限（谁能让它操作）。
- FR-4.3 Channel 接口抽象，微信等后续接入不改认知核。

### FR-5 呈现面 UI（本地）
- FR-5.1 显示"它当前在看什么"（被观察面缩略/标注，含当前视觉源）。
- FR-5.2 显示结论、动作日志、委派状态、当前任务。
- FR-5.3 self-view 与浮层默认不进入采集。

### FR-6 安全
- FR-6.1 动作确认门：危险动作（删除、支付、发送、关机等）需人工确认或显式授权模式。
- FR-6.2 全局急停：热键/语音一键停止一切动作与生成。
- FR-6.3 审计回放：记录"看到什么→决策→执行了什么动作/委派"，可回放复盘。
- FR-6.4 作用域边界：可限制作用范围（指定窗口/应用/目录）。
- FR-6.5 记忆隐私：常开感知的记忆有**保留期、脱敏、用户可查可删**；敏感画面（密码框等）可配置不入记忆。

### FR-7 记忆（Memory）
- FR-7.1 分层记忆：工作 / 情景 / 语义（长期）/ 程序性 四层（见 §6.5）。
- FR-7.2 巩固：后台把工作记忆里值得留存的经验提炼进情景/语义记忆。
- FR-7.3 检索回流：认知核可按当前情境检索相关记忆（向量+结构化）并注入上下文。
- FR-7.4 用户偏好与画像：记住用户习惯/偏好，跨会话生效。
- FR-7.5 记忆可查可删（与 FR-6.5 一致）。

### FR-8 调度与任务（Scheduler / Task）
- FR-8.1 定时任务：cron / 固定间隔 / 一次性提醒（如"每天9点汇报""10分钟后提醒我"）。
- FR-8.2 条件触发：注册"当感知到 X → 做 Y"的规则（如"看到构建失败就通知我"）。
- FR-8.3 触发即事件：到点/命中作为**内部事件**进认知核，与感知/channel 事件统一仲裁（见 §3.7）。
- FR-8.4 长时任务：目标/待办的跟踪、优先级、**断点续跑**（被打断/切源/重启后可恢复）。

### FR-9 技能（Skills / Tools）
- FR-9.1 技能注册表：可注册工具/技能（含内置：搜索、文件、通知等），走与 Effector 相同的能力协商。
- FR-9.2 **两类技能并存**（见 §6.5 与本文档"技能形态"讨论）：
  - **符号工具技能**：有文本 schema 的函数/API 调用（传统形态；复杂者可转交 Delegate）。
  - **感知-运动技能**：以"感知→行动"策略存在的过程性技能（如"登录这个 App""关掉这个弹窗"），来源于**程序性记忆/示范学习**，而非写死的 schema。
- FR-9.3 技能习得：成功的感知-行动轨迹可**巩固为可复用的感知-运动技能**（FR-7.2 的产物）。
- FR-9.4 技能同样过 Safety 层。

## 8. 非功能需求（NFR）

| 类别 | 要求 |
|---|---|
| **延迟** | "用户开口→开始响应" < 1s（受推理与 barge-in 影响）。**"屏幕变化→被感知"取决于采样帧率**：常开基线 ~1fps（最长约 1 帧周期 ≈1s），检测到变化时临时升帧（上探 10fps）可把感知延迟压到 ~100ms 级——不是恒定 <500ms，而是"静止省帧、变化升帧"（见 §6.6、§12.12）。 |
| **实时性** | 感知回路持续运行，不因单次推理或委派而阻塞采集。 |
| **可打断** | barge-in 中止延迟 < 300ms。 |
| **部署/算力** | Core 硬件无关，只需访问模型推理 API；推理服务可本地 GPU / 局域网 / 远程云 GPU（§3.6、§6.3）。"自托管"=模型服务归用户掌控，不强制本地 GPU。需给出显存/算力档位建议（见 §12）。 |
| **可扩展** | 新增执行器/委派/通道/channel 不改认知核（插件化）。 |
| **跨平台** | 兼容 **Windows / macOS / Linux / Android / iOS**：Core 一份平台无关代码，平台差异全部收敛在 Endpoint 层（采集/防回音/输入注入按 §6.4 矩阵各自实现）。移动端作瘦端远程连 Core。 |
| **安全** | 默认最小权限；危险动作默认需确认；全局急停始终可用；物理执行器分级授权。 |
| **可观测** | 全链路日志 + 动作/委派审计回放。 |
| **成本** | 常开感知按显著性动态采样，控制持续算力开销。 |

## 9. 执行器与委派抽象（重点：为物理世界"预留开放"）

**目标**：认知核只表达"想干什么（意图）"，不关心"用什么身体去干"。今天是键鼠，明天是机械臂、方向盘、家居开关，接口不变。

### 9.1 Effector 接口（概念）
- `capabilities()`：动作原语集合（`pointer.move`、`pointer.click`、`key.type`；未来 `touch.swipe`、`arm.grasp`、`vehicle.steer`、`home.setLight`）。
- `execute(action)`：执行一个动作原语，返回结果/失败。
- `constraints()`：物理/安全约束（作用域、速率、危险级别）。

### 9.2 v1 注册的执行器
- **KeyboardMouseEffector**：屏幕坐标 + 点击/键入。
- **VoiceEffector**：流式 TTS。
- **TextEffector**：文本输出到 UI/飞书。

### 9.3 执行器演进路径（终点是物理世界，绝非本地电脑）
本地电脑只是**第一步**：

```
本地电脑 → 云端电脑 → 手机 → 汽车 / 机器人 / 智能家居（物理世界·终极目标）
   ↑          ↑         ↑              ↑
 v1(键鼠)   近程扩展   触屏手势     待模型/硬件具备后
```

- **本地电脑**（v1）：`KeyboardMouseEffector`。
- **云端电脑**（近程）：共用键鼠+屏幕动作原语，走远程通道（RDP/VNC/云桌面流）。最低成本扩展点，验证"同一意图跨不同执行身体"。
  - **回音范围（按 §3.1 准则）**：若云端桌面**渲染到本地显示屏**（本地登录、本地窗口显示）→ **需按防回音处理**（本地浮层可能叠入被采集窗口）；若云端**不在本地屏渲染**（云端侧直接抓帧/无头）→ **不在回音解决范围**（观察面在云端、呈现面在本地，不同屏不叠加）。
- **手机**：触屏手势（见 §9.4.1）。
- **物理世界（终极目标）**：汽车（智能驾驶/智能座舱）、机器人（工业/家庭）、智能家居——彻底走进物理世界交互。

### 9.4 预留执行器（不实现，仅定义占位）
- `CloudPCEffector`（云端电脑）、`PhoneEffector`（手机）、`VehicleEffector`（汽车：智能驾驶/座舱）、`RobotEffector`（工业/家庭机器人）、`HomeEffector`（智能家居 IoT）。
- 引入时认知核**零改动**，只需注册新执行器 + 在 Safety 层加对应授权分级（物理执行器授权尤严）。

#### 9.4.1 PhoneEffector（手机触屏操控，§2.0 哲学的最佳例证）
- **感知方式与电脑同构**：看着**触屏手机屏幕**做决策，复用整套 GUI-Agent 感知→行动闭环，执行器从"键鼠"换成"触摸手势"。手机屏幕作为一路视频源接入（受单路+可切源约束）。
- **触摸动作原语**：`touch.tap`、`touch.long_press`、`touch.long_press_drag`（长按拖拽）、`touch.swipe`（左/右/上/下滑，含起止坐标与速度）、`touch.drag`、`touch.pinch`、`key.back/home`、文本输入。（与 FR-3.1T 一致）
- **意图—执行器解耦印证**：意图"打开那个 App / 返回上一页"不变，PhoneEffector 落成 tap / 左滑；同一意图在电脑上是鼠标点击。
- 落地通道（未来选型）：ADB（安卓）、iOS 自动化、云手机远程流。

### 9.5 意图—执行器解耦的收益
- 同一意图"点击那个红色按钮"：本地电脑=鼠标点击，云端电脑=远程键鼠事件，手机=触点，机器人=机械指——由执行器各自实现。

### 9.6 Cognitive Delegate（认知委派：向文本智能体外包符号任务）
弥补感知核在文本空间的短板（见 §3.5）。Delegate 与 Effector 平行：Effector 改变**外部环境**，Delegate 外包一个**认知子任务**并返回结构化结果，喂回工作记忆。

- **委派只有一种形态：结构化委派**（sub-agent / API / CLI / MCP，文本形式）。GUI 驱动工具**不是委派**（它是 §3.2/§9 的原生 GUI 操作，用自己的手脚），故 Delegate 不含 `gui` 模式。
- **Delegate 接口（概念）**：
  - `capabilities()`：任务类型（`code.write`、`code.fix`、`repo.refactor`、`longform.reason`）。
  - `dispatch(task, context) -> handle`：**异步**派发，立即返回句柄，不阻塞感知回路。
  - `poll(handle) / onResult(handle)`：回收结构化结果（patch/输出/日志/成败）。
- **v1 目标**：`CodexDelegate`、`ClaudeCodeDelegate`（经各自的 CLI/API/MCP 结构化调用）。
- **两条路径都保留（见 §3.5）**：让 ClaudeCode/Codex/Cursor 等工具干活，既可走**结构化委派**（本节，异步/非阻塞），也可走**屏幕操作该工具**（Effector，FR-3.1，像人一样开窗操作）。二者按场景选，不是互斥淘汰。
- **何时走屏幕操作**：工具无可用 API；或工具本质是 GUI（如 Cursor 这类 IDE）、像人一样用它更自然。代价是占用主回路（盯屏），非阻塞性弱于结构化委派。
- **非阻塞语义**：派发编码任务后感知回路继续运行；结果到达作为"感知事件"回流。
- **可切换/可扩展**：换编码智能体只需注册新 Delegate，认知核零改动。
- **安全**：Delegate 产生的动作同样过 Safety 层（§8）。

> 一句话：**委派 = 用别人的脑子（结构化交给文本智能体）；GUI 操作 = 用自己的手脚（Effector）。** Effector 是"手脚"，Delegate 是"可外包的专业大脑"，感知核是编排两者的指挥者。

## 10. Channel 抽象（飞书优先）
- **Channel 接口**：`receive()`（文/图/语音）、`send()`（文/语音）、`identity/permission`。
- v1：**飞书**（Lark）。预留：微信、Slack/Discord、Web、电话。

## 11. 路线图

> 你选择"四类核心场景都要"。这里按**依赖关系与验证价值**排序，每阶段可独立演示。

- **M0 · 骨架与防回音（地基）**
  感知面/呈现面解耦（`WDA_EXCLUDEFROMCAPTURE`）、感知事件总线、**Model Adapter 框架 + MiniCPM-o 与 JoyAI-VL 两个适配器（先打通视觉路径）**、本地呈现面 UI、单路+切源框架（M0 切源经 UI/热键触发，语音触发随 M1）。
  验证：屏幕流稳定进模型、不自我回环、UI 可见其所见、能一句话切到摄像头、**两款模型的视觉路径均跑通并可切换**（JoyAI 的 ASR/TTS 语音栈随 M1 接入，见 §6.6）。

- **M1 · 视觉+语音实时双模对话（场景 4）**
  麦克风流 + VAD + 语音输入；TTS；barge-in；持续常开感知 + 显著性驱动主动发声。
  验证："像人一样边看边聊、能被打断、会主动开口"。

- **M2 · 看屏 + 键鼠完成单任务（场景 1，GUI 闭环，看家能力）**
  KeyboardMouseEffector + Safety 层（确认门/急停/审计/作用域）。
  验证：给定具体任务（网页订票/整理文件夹）闭环完成。

- **M2.5 · 认知委派接入**
  CodexDelegate / ClaudeCodeDelegate（结构化委派：sub-agent/API/CLI/MCP，异步）。
  验证：把"写/改代码"外包给文本专家，期间感知不阻塞、结果回流后继续。

- **M2.8 · 通用 Agent 能力（记忆 + 调度 + 任务 + 技能）**
  分层记忆与巩固检索、定时/cron/条件触发调度（主动性的时间维）、长时任务断点续跑、技能注册表（符号工具 + 感知-运动技能雏形、示范学习）。
  验证：记住用户偏好跨会话生效、"每天9点汇报/看到构建失败就提醒"能触发、示范一次后能复用一个感知-运动技能。

## 12. 风险与开放问题（需后续拍板）

1. **推理算力档位与放置**：MiniCPM-o / JoyAI-VL 常开持续推理的显存/算力要求？推理服务放本地 GPU 还是远程/云 GPU API（Core 硬件无关，§3.6）？——本地 GPU 延迟最低、远程 GPU 省本地硬件但加网络延迟。**且远程/云 GPU 意味着屏幕流上行到外部、有隐私暴露，敏感场景应保留本地自托管推理**（与 FR-6.5 一致）。直接影响延迟与采样率预算。**当前最卡脖子。**
2. **显著性/主动性度量**：如何定义"值得主动开口/行动"的分数？初期用启发式（画面变化+语音停顿+任务进度），再迭代。
3. **多模态时间对齐**：视觉帧与语音片段如何在认知核对齐成"此刻发生了什么"？
4. **防回音极端情况**：全屏采集 + 呈现面必须可见时的兜底（虚拟显示器工程成本）。
5. **安全授权分级**：哪些动作永远需确认？物理执行器（车/机器人）如何分级授权与故障安全？
6. **成本上限**：常开感知的持续算力/电力，是否叠加"低功耗待命档"作为优化（已选"持续常开"）。
7. **底座模型能力边界**：实时视觉+语音双工在长时对话下的稳定性/上下文长度；**多路视频流何时可用**。
8. **委派与主循环协调**：多个异步 Delegate 并发时的结果合并、冲突与优先级。
9. **iOS 输入注入受限**：iOS 无公开跨应用自动化通路，"操作 iOS"近期只能靠 WebDriverAgent（开发态）或云真机；需确认 iOS 是"仅感知"还是要真正操控，以及可接受的落地方式。
10. **跨平台工程栈选型（部分已定：Python 主力）**：**Core 与桌面 Endpoint 以 Python 为主力语言**（与 MiniCPM-o/JoyAI 的 Python+vLLM/PyTorch 栈契合，也是团队熟悉语言）。**仅当 Python 无法覆盖某平台的采集/注入时（如 Android 需 Kotlin/Java 的 MediaProjection+AccessibilityService、iOS 需 Swift 的 ReplayKit/WebDriverAgent），该平台 Endpoint 才用原生语言实现，经统一实时流协议（WebRTC 类）接回 Python Core。** 待定：具体到 Android/iOS 时 Python 能覆盖到哪、原生壳多薄。
11. **两款模型的语音架构分歧落地**：MiniCPM-o 原生语音 vs JoyAI-VL 需外挂 Qwen3-ASR/Qwen3-TTS + background-agent（见 §6.6）。用 JoyAI 时需额外部署这些 vLLM 服务、并统一 barge-in/延迟表现——需确认部署与资源预算。
12. **视频帧率与显著性采样**：两款均以 ~1fps 决策为主（MiniCPM-o 可 10fps）。常开下如何在"够用的实时性"与"算力成本"间取采样策略（静止降帧/变化升帧）。
13. **许可合规复核**：两款权重 HF 卡均标 Apache-2.0 可商用；但 MiniCPM-o repo 是否有附加 Model License、JoyAI 的 `Preview` 状态与其依赖的 Qwen3-ASR/TTS 许可，需商用前逐一核实（见 §1.4）。
14. **坐标接地（grounding）精度**：GUI 闭环（看家能力，M2）依赖模型输出**准确的点击坐标**，而 VLM 的像素级 grounding 是公认难点。需评测 MiniCPM-o/JoyAI-VL 的 grounding 精度，必要时叠加元素检测/OCR 辅助定位与动作后校验（FR-3.6）。**与 §12.1 并列为 M2 关键风险。**

## 13. 术语表
- **Afferent / Efferent**：传入（感知）/ 传出（行动）通路，借自神经科学。
- **Cognition Core（认知核）**：编排逻辑（感知总线+工作记忆+意图/委派编排），**硬件无关**，只需访问推理 API，产出意图。
- **Model Inference（模型推理服务）**：吃 GPU 的多模态模型前向计算，以 API 暴露；宿主可本地/局域网/云 GPU。与 Core 解耦。
- **Model Adapter（模型适配器）**：把某款模型封装到 Core 统一接口后的适配层，声明其实时能力剖面；v1 完全兼容 MiniCPM-o 与 JoyAI-VL，未来加模型只加 Adapter。
- **Endpoint（端）**：每平台薄适配器，负责采集+执行+权限；移动端为远程瘦端。
- **Observation / Presentation Surface（被观察面/呈现面）**：解决"视觉回音"的核心抽象。
- **Effector（执行器）**：把意图落地为动作的"手脚"，可插拔（键鼠→车/机器人/家居）。
- **Delegate（认知委派）**：把符号/文本子任务**结构化**外包给另一个文本智能体（sub-agent/API/MCP）的"可外包的专业大脑"（用别人的脑子）。**只有结构化一种形态**；GUI 操作工具属 Effector，不是委派。
- **Source Switching（源切换）**：单路视频流在屏幕/摄像头等源之间运行时热切换。
- **控制指令 vs 行动意图**：前者调节系统自身（切源/停止），后者作用于外部环境。
- **Barge-in**：用户打断，Agent 立即停下倾听。
- **Salience（显著性）**：驱动"主动交互"的内部注意力评分。
